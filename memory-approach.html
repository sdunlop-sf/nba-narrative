The core insight
💡
The system's value is in the memory, not the brain.
An LLM can make a decent decision on day 1 with zero history. But on month 6, a system with a learning loop is making decisions that an LLM alone can never make — because it knows what actually worked for customers like this one.
Why this matters now
Every vendor is building a smarter brain. Rules engines gave way to ML models, ML models are giving way to LLMs and agents. The brain is commoditizing — reasoning capability is approaching parity across platforms. What compounds is the memory: the accumulating record of decisions made, outcomes observed, constraints learned, and relationships understood. The system that maintains this memory makes every future decision better. The system that only has a brain starts from scratch every time.
The reduction
In a future where reasoning is free and all data lives in one place, decisioning reduces to two things: memory (what happened, what worked, what constraints apply) and value (what matters, measured how). The infrastructure that maintains both — and connects them through a learning loop — is the entire product. Everything else is commoditized compute.
01
Four Primitives
Instead of the traditional strategy-interaction-treatment model, a memory-first architecture builds on four primitives that separate the durable assets from the swappable reasoning layer.
Primitive 1
Relationship State
Not "customer data" but the living model of each customer relationship. Current state, trajectory (getting better or worse), decision history, sentiment, constraints, value. The question the system answers isn't "what are this customer's attributes?" but "what is the state of this relationship and where is it heading?"
CRM + Data Cloud + Decision Journal fused into a queryable state per customer
Primitive 2
Action Space
Everything you could do, constrained by everything you shouldn't. Policy, budget, eligibility, fatigue, legal constraints, channel availability — all evaluated before the reasoning engine sees a single action. The brain operates within pre-constrained boundaries. It never evaluates an illegal action.
Policy is the wall of the space, not a post-filter on the output
Primitive 3
Value Function
What are we optimizing, measured how? Not hardcoded to propensity (most likely to accept) or engagement (most likely to click). Composable: a bank weights 40% CLV + 40% compliance + 20% satisfaction. A telco weights 80% CLV + 20% churn risk. The value function IS the business strategy, expressed as math.
Industry templates ship a value function, not just a catalogue
Primitive 4
Learning Loop
This is the actual product. Every decision produces an outcome. Every outcome updates the relationship state and refines the value function. The decision journal, outcome attribution, adaptive weights, model retraining — this is the infrastructure that compounds. Month 6 should be measurably smarter than month 1.
The brain is pluggable. The loop is the moat.
The reasoning engine is deliberately absent from the primitives
The brain — rules, ML, LLM, hybrid — queries the relationship state, operates within the action space, optimizes the value function, and feeds the learning loop. It's the consumer of all four primitives, not one of them. Today's brain is deterministic scoring with LLM re-rank. Tomorrow's could be a full LLM planner, a contextual bandit, or something that doesn't exist yet. Because it operates against well-defined interfaces, the swap is clean.

03
What Decisioning Actually Requires
The "fluid" characterisation works for one-shot reasoning. In practice, operationalising decisions at scale needs four things that a stateless agent cannot provide.
⏳
Decisions managed across time
Cooldown windows, fatigue scoring, contact policy — you can't ask "what's the best action now?" without knowing what you already did yesterday. An LLM with a prompt doesn't track that a customer got 3 emails this week and the marginal value of a 4th is negative. That's state. That's a system.
🛤️
Rails, not just reasoning
The LLM composes a novel treatment, but it runs through budget caps, contact policy, and confidence thresholds before it touches a customer. In regulated industries, the rails ARE the product. Governance, auditability, and deterministic enforcement aren't features — they're the foundation.
📈
Closed-loop learning
Value optimization requires feedback loops that compound over time. Adaptive weights, attribution, retraining triggers. The "infinite software" thesis can build a decisioning agent in an afternoon. It cannot build the infrastructure that makes that agent better in month 3 than month 1.
📋
The action catalogue
"Decisioning isn't owned by anyone" is true for the reasoning step. But someone has to maintain the catalogue of actions, eligibility rules, expected values, budget constraints, channel routing, and treatments. That's operational infrastructure, not a prompt.
The one-sentence version
Everyone builds a smarter brain and bolts memory on as reporting. Nobody builds the memory first and treats the brain as pluggable.
04
Where the Market Is Today
Pieces exist. Nobody has unified them with relationship state at the center and the learning loop as the primary asset.
Traditional CDH

Adaptive models, action management, contact policy, next-best-action. Brain-centric: the scoring engine is the center of gravity. Relationship context assembled at decision time, not maintained. Learning loop exists as a feature of scoring, not the architectural foundation.

Pre-cloud, pre-LLM infrastructure
Journey Optimizers

Offer decisioning with eligibility and capping. Journey-centric: answers "which offer at this journey step" not "what's the optimal next action in this relationship's trajectory." Learning loop optimizes engagement metrics (clicks, opens) not relationship value.

Optimizes tactics, not strategy
Feature Stores & Data Platforms

Closest to "relationship state as infrastructure" — maintain point-in-time feature vectors per entity. Plumbing, not a product. No decisioning layer, no action space, no value function, no learning loop. But strategically dangerous: if the decision memory lives here instead of in the engagement platform, the engine becomes a commodity and the data layer owner has the leverage.

Data layer — today's plumbing, tomorrow's moat competitor
Vibe-Coded Alternatives

An LLM can generate a scoring engine, an action catalogue, and an arbitration layer in an afternoon. The code is not a moat. What it cannot generate: 12 months of accumulated decision outcomes, a calibrated value function, or learned contact policy constraints. In a world where reasoning is free, the only defensible asset is the accumulated memory.

Commoditizes the engine, validates the memory thesis
What nobody has
Composable value function
Everyone hardcodes their optimization objective. Nobody lets you configure "40% CLV + 40% compliance + 20% satisfaction" and swap that per industry.
Decision journal as primary data model
Everyone treats decisions as output (logs). Nobody treats the journal as the input — the accumulating asset that makes every future decision better.
Pre-constrained action space
Everyone post-filters: score everything, then check policy. Nobody defines the legal action space first and hands a constrained set to the reasoning engine.
Explicitly swappable reasoning
Every vendor's brain is welded in. Nobody has separated state/space/value/loop from the brain cleanly enough to make the reasoning engine a pluggable layer.
05
Time as a First-Class Dimension
Not "what's the best action now?" but "what's the best next action in this relationship's trajectory?"

The system knows that sending a retention offer, then an upsell, then a satisfaction survey has a different expected value than the reverse order — because the learning loop has seen both sequences play out. This is sequence-aware decisioning. It's impossible without the decision journal and the learning loop working together.

The compounding effect
Day 1: The engine scores on historical CRM data. A good starting point. An LLM alone can do roughly this well with the right prompt and context.

Month 1: The learning loop has observed 10,000 outcomes. Adaptive weights have shifted. The engine knows which actions work for which segments — not from training data, but from its own decisions.

Month 6: The decision journal contains sequence patterns. The value function has been calibrated against real attribution data. The system has observed which action sequences compound relationship value and which erode it. No stateless agent can match this.

Year 1: The memory is the moat. A competitor can replicate the reasoning engine in a weekend. They cannot replicate 12 months of compounding institutional memory. The switching cost isn't the software — it's the loss of everything the system has learned.
06
Deciding vs. Serving
The Decisioning Hub decides what to do. It doesn't deliver the experience. That separation is the architecture.
Decisioning Hub (batch, promotion-level)
Evaluates relationship state, constrains the action space, scores the value function, selects the best promotion
Writes latent decisions to Data Cloud — pre-computed, cached, ready for downstream consumption
Runs on its own schedule (batch, segment-triggered, event-driven). No Apex in the hot path.
Records everything in the decision journal. Owns the learning loop.
vs
Serving Layer (real-time, offer/treatment-level)
Reads latent decisions from Data Cloud. Selects the best offer, treatment, and channel for the moment.
Optimises in real-time: which creative, which channel, which time. This is what SP / Personalisation does natively.
Operates at whatever concurrency the channel requires. No governor limits in the serving layer.
Returns channel-level outcomes to the Hub for the learning loop.
The clean boundary
The Hub owns the what (which promotion, for whom, why). The serving layer owns the when and where (which offer, which treatment, which channel, right now). Without the Hub, the serving layer optimises tactics with no relationship intelligence. With the Hub, it serves offers backed by compounding memory. That's the upgrade story.
07
Closing the Loop for Real
The learning loop only compounds if outcomes reflect actual business results, not just agent clicks.

"Accepted" is a proxy metric — it means someone in the process approved the recommendation. It doesn't mean the customer bought the product, renewed the policy, or stayed another year. The gap between decision acceptance and business outcome is where attribution lives, and it's where most decisioning systems quietly stop.

✅
What we capture today
Agent or system accepts/rejects the recommendation. CampaignMember status writes back to Data Cloud. Einstein Studio retrains on this signal. This is necessary but not sufficient.
🎯
What the loop actually needs
Match the decision to the downstream business event. Did the retention offer lead to a renewal? Did the upsell become an Opportunity? An attribution window that joins NBADecision to CRM transactions — configurable per module, closed by batch.
Why this is the next frontier
Adaptive weights and model retraining are only as good as the signal they train on. If "accepted" is the signal, the system learns to optimise for agent agreement. If "renewed within 90 days" is the signal, the system learns to optimise for business outcomes. The quality of the learning loop determines the quality of every future decision.
08
Why This Belongs on CRM
The primitives map naturally to platform capabilities that already exist.
Relationship State
CRM + Data Cloud is already the system of record for customer relationships. Contact, Account, Opportunity, Case — fused with Data Cloud's unified profile and identity resolution. The decision journal lives alongside the relationship it's about.
Data Cloud + CRM = relationship state infrastructure
Action Space
Real Time Offer Management provides the promotion catalogue with eligibility, margin, and content natively. Governance, permission sets, and sharing rules are what CRM was built for. The rails layer inherits decades of enterprise-grade infrastructure.
RTOM + governance + sharing = pre-constrained action space
Value Function
Einstein Studio models + Data Cloud calculated insights provide the signals. The value function composes them with configurable weights. Industry templates ship pre-tuned objective functions — banking weights compliance, telco weights churn risk.
Einstein Studio + Data Cloud CIs = composable signals
Learning Loop
Outcomes write back to CRM and Data Cloud. CIs aggregate decision patterns. Einstein Studio retrains on outcome data. Agentforce agents reason about the system itself — not just per-decision, but about whether the decisions are collectively working.
CRM outcomes + Data Cloud + Einstein Studio = closed loop
The strategic position
This space has been historically ceded to specialist vendors. The platform is better positioned for it now than they are. Data Cloud gives us the semantic layer. Agentforce gives us the reasoning. CRM gives us the business process and governance that make it enterprise-safe. The gap is the orchestration layer that ties them together — the memory-first decisioning system that turns platform capabilities into compounding relationship intelligence.
