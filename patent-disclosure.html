1. Problem Statement

Current NBA decisioning systems operate on a fixed catalogue of pre-defined actions. When no action meets eligibility, scoring, or compliance thresholds, the system returns nothing — a silent failure representing lost revenue and missed customer engagement.

This is the dominant pattern in both Salesforce's existing NBA capabilities and competitor platforms such as Pega CDH and Adobe Target. The problem compounds over time: as customer segments evolve and business conditions shift, the static catalogue becomes increasingly misaligned with what customers actually need, requiring manual intervention from business users to create new actions — a slow, reactive process.

Additionally, when organisations attempt to use large language models (LLMs) to generate customer-facing treatments dynamically, they face a compliance and governance gap: LLM-generated content cannot be trusted to respect business constraints (budget caps, contact frequency policies, channel eligibility, regulatory suppressions) without a deterministic enforcement layer.

The Binary Problem

Organisations face a forced choice: fully deterministic (safe but rigid — the catalogue can't adapt) or fully generative (flexible but ungoverned — the LLM can't be trusted to respect compliance). There is no production-safe middle ground. This invention provides one.

2. Solution Overview

A composition layer that sits inside an existing deterministic arbitration pipeline. When no catalogued action is eligible, an LLM composes a novel treatment — then that treatment is validated by the same deterministic compliance checks that catalogued actions pass through.

Treatments that pass constitutional-rails validation are surfaced for human approval. Approved treatments that generate positive customer outcomes can be promoted to first-class catalogue entries, creating a self-improving action catalogue that grows organically without manual authoring.

3. Detailed Architecture

3.1 Deterministic Arbitration (existing foundation)

For a given customer contact, the system:

  1. Queries active NBA modules (NBA_Module__c)
  2. Retrieves a propensity score from an Einstein Studio predictive model via a scoring Flow
  3. Filters a catalogue of pre-defined actions (NBAAction__c) through eligibility rules: score thresholds, date windows, budget caps, segment membership, channel frequency policies, global suppressions
  4. If one or more actions pass all filters, the highest-weighted action is selected and served — no composition is invoked

3.2 Composition Trigger

When deterministic arbitration produces no eligible action (all catalogued actions filtered out), the system invokes the composition path instead of writing a null-action decision. The trigger passes the full decision context to the LLM, including:

  • Customer propensity score and segment memberships
  • Every action that was considered and the specific reason each was dropped (score below threshold, budget exhausted, channel frequency cap, segment mismatch, etc.)
  • Customer's recent decision history
  • Active experiment constraints (if any)
  • Module-level business context (outcome type, priority)

3.3 LLM Treatment Composition

An Agentforce-hosted LLM agent receives the decision context and composes a novel treatment — a customer action that does not exist in the pre-defined catalogue. The composed treatment includes:

4. Constitutional Rails Validation

Core Innovation

The composed treatment is validated by the identical deterministic compliance methods that catalogued actions pass through. There is no separate "LLM compliance path." The composed treatment is marshalled into the same data structure and run through the same Apex validation methods.

The validation method ComposeTreatment.validateAgainstRails() runs the following checks against the composed treatment:

  1. Budget cap: If the treatment includes an offer value, validate against the module's budget ceiling. Uses the same Budget_Used__c / Max_Offer_Budget__c comparison with FOR UPDATE row locking as catalogued actions.
  2. Contact frequency policy: The treatment's channel is checked against the customer's recent contact history using the same pre-queried channel maps (latest contact datetime, 7-day count) and NBA_Channel_Policy__c rules.
  3. Global suppression: Customer checked against active NBA_Global_Suppression__c records (opt-outs, regulatory holds, segment-based blocks).
  4. Segment eligibility: If the treatment targets a specific segment, customer membership is verified against the eligibility segment list.
  5. Cross-module cooldown: Checked against the configurable cooldown window to prevent over-contacting across modules.

Each rail evaluation is logged to a Rails_Checked__c JSON field with pass/fail status, the value checked, and the threshold applied. If any rail fails, the composition is rejected before human review.

Non-Negotiability Principle

The LLM agent cannot override, negotiate with, or circumvent the rails layer. Rails enforcement is deterministic Apex code, not prompted. Humans never see compositions that breached a rail — a failed-rail composition is a rejected error, logged for diagnostics, not a "composition pending review." The rails are constitutional: set by compliance/legal/finance stakeholders, enforced by deterministic code, not subject to LLM reasoning.

Rails-as-Code Design Properties

  1. Deterministic — not prompted. Actual Apex code that runs with the same logic regardless of the LLM's output.
  2. Auditable — logs every rail evaluated, value checked, pass/fail result to Rails_Checked__c.
  3. Non-negotiable — if any rail fails, the composition is rejected BEFORE human review. Agent cannot "convince" the rails layer.
  4. Unified — any future compliance rule added to the catalogued action path automatically applies to composed treatments without additional implementation.

5. Human Approval Queue

Composed treatments that pass all rails checks are written with status pending_approval and surfaced in a Composition Queue tab within the Decisioning Hub. The reviewer sees:

  • Full decision context (score, segments, dropped actions with reasons)
  • The composed treatment (channel, offer, content draft)
  • Composition rationale (LLM's natural language reasoning)
  • Rails validation audit trail (every check, pass/fail)
  • Agent confidence score

Available actions:

  • Approve & Send — triggers execution through the standard channel delivery
  • Reject — logged with rationale for future model learning
  • Edit & Approve — human modifies the treatment before approving

6. Evidence-Gated Catalogue Promotion

The composition-to-catalogue loop is the mechanism by which the system self-improves without manual action authoring.

Approved compositions that are executed follow the same outcome tracking pipeline as catalogued actions — customer responses are recorded as NBADecision__c outcomes and fed back into Einstein Studio for model retraining.

Composed treatments that are approved and subsequently generate positive customer outcomes become candidates for promotion to first-class catalogue entries. A promoted composition becomes a permanent NBAAction__c record with:

  • Score threshold derived from the propensity score at which it was originally composed
  • Channel, offer value, and eligibility rules derived from the composition context
  • Priority weight informed by the outcome acceptance rate

The Closed Loop

The LLM generates hypotheses. Deterministic rails enforce boundaries. A human gates quality. Real-world outcome data gates promotion. The catalogue grows organically from proven compositions — the system self-improves without a product manager manually designing new actions.

7. Null-Score Suppression Pattern

A supporting innovation within the arbitration engine: when the Einstein Studio predictive model is unavailable or untrained, the system writes a deliberate "restraint" decision — a null-action decision with reason "Model not connected" — rather than using a fabricated fallback score or failing silently.

This restraint decision is:

  • Tracked as a first-class outcome in the decision audit trail
  • Fed back into Einstein Studio training as a "do not contact" signal
  • Visible in the Decisioning Hub UI with a clear explanation

This ensures the system never makes decisions based on fabricated data, and the model learns from the system's honest uncertainty rather than from actions taken on fake scores.

8. Key Differentiators

8.1 Single-Path Constitutional Validation

LLM-composed treatments are validated by the identical deterministic compliance methods as pre-defined catalogue actions. No separate "LLM validation path" exists. Any compliance rule added to the catalogued action path automatically applies to composed treatments without additional implementation. This eliminates governance drift — the risk inherent in systems that maintain parallel validation paths for human-authored and machine-generated content.

8.2 Evidence-Gated Composition-to-Catalogue Promotion

The closed-loop promotion pattern — compose → validate against rails → human approval → execute → measure outcome → promote to catalogue — creates a self-improving decisioning system. The catalogue evolves based on treatments that have been validated at every layer: deterministic compliance, human judgment, and real-world customer outcomes. No manual action authoring is required for the catalogue to grow.
