What It Is

The NBA Decisioning Accelerator is an unlocked Salesforce package that delivers a production-grade Next Best Action decisioning engine entirely within the Salesforce platform. It combines a modular, bring-your-own-model ML architecture for predictive scoring with a deterministic arbitration engine for action selection, and an Agentforce agent for intelligent adjudication of complex decisions — all without external dependencies.

Key Principle
No LLM in the hot path. ML models score, deterministic logic arbitrates, and Agentforce reasons only when human review is required. This keeps latency low, costs predictable, and decisions auditable.

The ML Pipeline

The accelerator is built on a modular, bring-your-own-model architecture. Each decisioning module can be backed by its own scoring source — the arbitration engine doesn't care where the score comes from, only that it receives one.

Scoring Sources

Source How It Works When to Use
Einstein Studio Binary Classification model trained on Data Cloud DMOs. Scores are retrieved via the Einstein prediction Flow action. Standard use case — churn, propensity to buy, renewal risk. Training data lives in Data Cloud.
External Model (BYOM) Any model accessible via a Named Credential + REST endpoint. The module stores the credential name and endpoint path; the engine calls it at runtime. Teams with existing ML platforms (SageMaker, Vertex AI, Azure ML, custom APIs) who want to keep their models where they are.
Data Cloud Score Field Pre-computed score stored on a Data Cloud DMO field. No runtime callout needed — the engine reads the value directly. High-frequency decisioning where latency matters, or when scores are batch-computed externally and synced to Data Cloud.

Multi-Module Architecture

  • One module per outcome — Churn, Upsell, Renewal, Winback, etc. Each module is independently configurable with its own model source, actions, and thresholds
  • Cross-module arbitration — When multiple modules fire for the same contact, the engine evaluates actions across all modules and applies cross-module cooldowns to prevent over-contact
  • Independent lifecycle — Modules can be activated, paused, or reconfigured without affecting other modules

Einstein Studio Model Details

  • Model type: Binary Classification (extensible to other types as Einstein Studio evolves)
  • Training data source: CampaignMember Data Model Object (DMO) in Data Cloud — captures outcome signals (accepted/declined) from prior decisions
  • Scoring: Each contact receives an outcome probability score (0–1) that feeds directly into the arbitration engine
  • Retraining: Automated retraining scheduler monitors outcome volume — when the threshold is met, a new model version is trained and the AUC metric is tracked

Training Feedback Loop

  • Decision approved → Campaign + CampaignMember created (Status: Sent)
  • Customer responds → CampaignMember status updated (Responded → Accepted/Declined)
  • CampaignMember syncs to Data Cloud → becomes training signal for next model version
  • This creates a continuously improving model that learns from real decision outcomes

The Arbitration Engine

A deterministic, Flow-based engine that evaluates every eligible action for a given contact and selects the optimal treatment.

How It Works

  1. Module selection — Active modules (e.g. Churn) define which outcome type to evaluate
  2. Score retrieval — Einstein Studio model score for the contact is fetched (or simulated in demo mode)
  3. Action eligibility — Each NBA Action is checked against:
    • Minimum score threshold (does the contact's score warrant this action?)
    • Segment eligibility and exclusions (Data Cloud segment membership)
    • Channel frequency caps (email: 48h cooldown, push: 24h, outbound call: 168h)
    • Global suppressions (contact-level or segment-level opt-outs)
    • Budget and date-range constraints
  4. Weighted scoring — Each eligible action's score is calculated: churn_score x priority_weight
  5. Tiering decision — Based on score proximity between the top two actions

Decision Tiering

Tier Condition Handling Expected Volume
Tier 1 Clear winner — top action's weighted score exceeds the runner-up by more than the proximity threshold Auto-adjudicated. No human review needed. Decision recorded and action executed immediately. ~80% of decisions
Tier 2 Competing signals — top two actions score within the proximity threshold (default: 0.050) Routed to Agentforce for AI-assisted adjudication with human approval. ~15-20% of decisions

Agentic Adjudication (Tier 2)

When the arbitration engine identifies a close call, the decision is escalated to an Agentforce agent that reasons over the full context and produces a recommendation for human review.

What the Agent Does

  1. Receives full decision context — contact profile, model score, all eligible actions with weighted scores, channel history, offer values
  2. Reasons about the best treatment — weighs channel effectiveness, offer value, score proximity, and contact context to recommend the optimal action
  3. Drafts outreach content — generates channel-appropriate content:
    • Outbound call: Call guide with objective, key talking points, and next steps
    • Email: Subject line and personalised email body
    • Push notification: Short, targeted notification text
  4. Human approves or overrides — the analyst reviews the recommendation and either approves (saving the agent's rationale and draft) or overrides with their own reasoning
Why Agentforce, Not a Flow
Tier 2 decisions involve ambiguous trade-offs (e.g. a high-value email offer vs. a personalised call with a relationship manager) where deterministic rules can't capture the nuance. The Agentforce agent applies reasoning to weigh these trade-offs and generates contextual outreach — something a Flow cannot do. Critically, the agent never acts autonomously; it recommends and drafts, but a human makes the final call.

Technical Implementation

  • API: Agentforce Agent Runtime API at api.salesforce.com
  • Authentication: OAuth 2.0 Client Credentials flow with JWT access tokens — scoped to chatbot_api, sfap_api, and api
  • Agent type: EinsteinServiceAgent configured via Agent Builder
  • Integration: Apex controller manages session lifecycle and message routing; LWC provides the review/approval interface
  • Audit trail: Agent rationale and drafted content are persisted on the NBADecision__c record

The Cockpit UI

A purpose-built Lightning app ("Decisioning Hub") gives analysts and managers full visibility and control.

  • Overview dashboard — Decision volume, tier split, model health, outcome rates
  • NBA Catalogue — Configure actions, channels, thresholds, offers, eligibility segments
  • Contact Policy — Channel frequency caps and cooldown windows
  • Live Simulation — Test arbitration against any contact in real time, see exactly which actions qualify and why
  • Tier 2 Queue — Review queue for decisions awaiting adjudication
  • Management Agent — Conversational interface for adjudication, status checks, and system management
  • Data Cloud Setup — Guided configuration wizard for Data Cloud bundles, segments, and model training

Competitive Positioning

The accelerator demonstrates that Salesforce can deliver a Pega-style decisioning capability as an embedded platform feature rather than a standalone product. Key differentiators:

  • Native platform: No external infrastructure, no data egress, no additional vendor contracts
  • Bring your own model: Works with Einstein Studio, external ML platforms (SageMaker, Vertex AI, Azure ML), or pre-computed Data Cloud scores — customers aren't locked into a single ML stack
  • Multi-module: One engine handles churn, upsell, renewal, winback, and any future outcome type — each with its own model, actions, and thresholds
  • Agentforce reasoning: AI-assisted adjudication that goes beyond rule-based decisioning, with full human oversight
  • Unlocked package: Deployable to any Enterprise org via standard package install + guided setup
  • Closed-loop learning: Decision outcomes feed back into model retraining automatically
