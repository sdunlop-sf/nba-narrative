The Trigger
What fires the flow and starts the decisioning process
Primary trigger — Data Cloud Segment Membership Change

You define a real-time segment in Data Cloud — e.g. "customer hasn't ordered a new SKU in 60 days and has 3+ tap handles active." The moment a customer enters that segment, Data Cloud fires a Data Action — essentially a platform event — into Salesforce. That platform event is what kicks off the Flow. The customer is processed immediately, in real time, without waiting for a batch run.

🕐
Scheduled Flow
Daily batch run across all accounts. Useful for morning rep briefings — every rep starts the day with fresh NBA priorities across their territory.
📝
CRM Record Change
Contract renewal date crosses a threshold. Opportunity stage changes. Account health score drops. Any Salesforce record update can kick the flow.
🛒
Inbound Signal
Customer logs into SAP Commerce portal. Clicks a product category. Abandons a cart. An API call from an external system fires the platform event.

03
Arbitration
Multiple NBAs have passed. Pick the one that gets executed.

After the loop, you have a shortlist — typically 2–5 NBAs that are eligible and scored above threshold. Arbitration logic picks the winner. The output is a single NBA (or a ranked list of 2–3 for rep discretion).

🏆
Highest score wins
The simplest approach. Einstein's top-scoring NBA gets executed. No overrides needed in most cases.
📈
Priority rules
Revenue-generating NBAs rank above retention NBAs when scores are within 0.1 of each other. Configurable per business context.
😴
Fatigue rules
If this customer received the top-scoring NBA on the last rep visit, serve the second-ranked. Prevents reps delivering the same message every call.
📡
Channel constraints
If the trigger was a portal login, only surface portal-deliverable NBAs. Rep-only NBAs are filtered out for digital triggers.
04
Execute
The winning NBA gets delivered. Channel depends on the trigger.
👤
Rep Visit Brief

Flow creates or updates a Salesforce Task on the account. Includes the NBA detail, talking points, and any linked Promotion & Offer. Rep sees it in their CRM view before the visit.

🖥️
Portal Personalisation

Flow calls Salesforce Personalisation via API. Recommendation renders in the SAP Commerce portal in real time when the customer is logged in. JS tag only — no structural rebuild.

✉️
Email via Marketing Cloud

Flow triggers a Marketing Cloud journey entry event. Email sends with the offer baked in, referencing the live Promotion & Offer record for eligibility and mechanics.

⚡

All three in parallel: Flow branches at execution and fires all three channels simultaneously — rep gets the Task, customer gets the portal recommendation, email triggers in the background. One NBA decision, three coordinated touchpoints.

Feedback Loop
What makes it a learning system rather than a static rules engine

Rep marks Task complete. Customer responds to offer. Portal click results in order.

☁️
Written to Data Cloud

→
🧠
Einstein retrains
Weekly or fortnightly retrain on new outcome signals. Scores improve over time.
Why this matters
On day one, Einstein is scoring based on historical CRM and ERP data — it's a good starting point but not yet tuned to your specific reps, regions, and customers. Every accepted NBA, every responded offer, every portal conversion is a new training signal. After 3–4 months of live operation the model has seen thousands of real outcomes and the scores become meaningfully more accurate than any rules-based approach could deliver.

What It Takes to Build for Real
The Flow wiring is the fast part — here's where the real effort sits
4–6w
Unified Profile in Data Cloud
Mapping source systems (CRM, ERP, Commerce) into clean DMOs. This is the longest dependency — everything else sits on top of it. If data is already clean in the ERP, compress to 3 weeks.
2–4w
Train Einstein Studio
Models exist in the org. Need historical outcome data: what NBAs were offered, which converted. If CRM win/loss data is reliable, this is straightforward. Start with rules-based scoring if it's patchy.
2–3w
Master Flow + NBA Catalogue
The actual Flow orchestration once the data foundation is in place. The catalogue object needs commercial alignment on what the NBAs are — that's a stakeholder conversation, not a technical one.
