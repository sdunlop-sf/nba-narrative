Architecture
⚙️
Deterministic when it can be. Reasoning when it should be. Creative when it has to be.
The engine runs in four modes depending on decision difficulty. Most decisions never leave Tier 1. The harder the call, the more reasoning gets applied — and the rails hold in every mode.
Tier 1
~85%
Deterministic Arbitration
Weighted score per eligible action — module_score × priority × response_weight. Winner is clear, rails pass, decision stamped. Sub-second. No LLM in the hot path.
Fires when: clear score gap, rails pass
Tier 1.5
10–15%
Reasoning Re-Rank
Top two scores within Reasoning_Gap_Threshold. Claude Haiku re-ranks the shortlist with context. 3s CPU circuit breaker. Parse failures fall through silently to the deterministic winner.
Fires when: narrow-gap or low confidence
Tier 2
2–5%
Human-in-the-Loop
Drafter agent writes a rationale and a recommended action into the Decision record. Slack approval card + Review Queue in the Hub. Human picks from eligible actions or dismisses. Siblings in the same run supersede.
Fires when: still ambiguous after Tier 1.5
Compose
<1%
Novel Treatment
No catalogued action clears threshold. Composer async-drafts a new treatment — channel, offer, content — then re-runs every rail deterministically. Rails-checked drafts land in the unified Review Queue for approval.
Fires when: catalogue has a gap
🔀

How tiers are selected: Tier 1 runs first — always. If the score gap is narrow or the winner is low-confidence, Tier 1.5 re-ranks. If the picture is still ambiguous, Tier 2 escalates to a human with an agent-drafted rationale. If nothing in the catalogue fits, composition kicks in. Every tier writes the same audit trace to Decision_Trace__c — schema-versioned JSON with eligibility, scores, reasoning, and rails evaluated.

The Trigger
What fires arbitration — configured per module as manual, scheduled, or real-time
Primary trigger — Data Cloud real-time segment activation

Every module has its own NBA_Module_Trigger__c rows. In real_time mode, a Data Cloud Real-Time Segment publishes a generic NBA_Decision_Trigger__e platform event. A handler Flow resolves the Contact, stamps the module + trigger type, and invokes the arbitration engine. Sub-second from segment entry to Decision record.

🕐
Scheduled
NBASegmentTriggerSchedulable — hourly Apex. Walks every active scheduled trigger, evaluates its segment, and fires arbitration for each member. Useful for morning rep briefings and nightly nurture sweeps.
🔌
Real-Time API
POST /services/apexrest/nba/v1/decide — upstream systems get a decision synchronously. Verbose mode returns the full trace. Same response contract as the MCP tool.
🤖
Manual / Agent
Management Agent, Live Simulator, or an MCP tool call via InvokeArbitration (global @InvocableMethod). Used for testing, rep-initiated decisions, and agent-driven orchestration.
🛡️
The Rails
Constitutional checks. Enforced in Apex. Never bypassed by reasoning.

Every decision — Tier 1, Tier 1.5, Tier 2, composed — runs through the same rails. They exist precisely because agents can be wrong. Agents cannot convince the rails layer. Humans configure them. Compliance, finance, and operations own them.

Rails evaluated per decision · persisted to Decision_Trace__c
Global suppression
Opted-out, vulnerable customer, regulatory block. Writes a null_action row with the specific reason. The unbreakable floor.
Constitutional
Channel policy
Per-channel cadence and cooldown (e.g. email 2/48h, SMS 1/72h). Bypassable only by Urgent or Critical flagged actions with a stamped bypass reason.
Configurable
Cross-module cooldown
Same contact doesn't get hit by multiple modules within N hours. Only Critical actions (fraud, security, regulatory) punch through.
Configurable
Eligibility
Per-action segment / exclusion / product holding / date bounds. Actions failing eligibility are dropped before scoring.
Per action
Budget cap
Each action has an offer budget. Budget_Used__c increments on every accepted outcome; arbitration drops the action the moment the cap is hit.
Per action
Score threshold
Per-action MinScoreThreshold__c. A low-confidence winner is the signal that either composition should fire (catalogue gap) or restraint is the right answer.
Per action

Restraint is a first-class outcome. When every action fails a rail, the engine writes a null_action row with the specific suppression reason rather than silently returning nothing. Einstein Studio trains on these rows too — "we chose not to speak to this contact" is a training signal, not an absence of one.

Escalation ladder — when Tier 1 isn't enough
TIER 1.5
Re-rank
Top two scores within Reasoning_Gap_Threshold__c. Tier1ReasoningArbiter dispatches the shortlist + full context to Claude Haiku via Einstein Trust Layer. JSON returned, validated, winner swapped if the model has clear contextual rationale. Hard 3s CPU budget. Parse failures are silent fallbacks — the deterministic winner stands.
TIER 2
Human review
Still ambiguous or flagged for mandatory review. Tier2AgentDrafterQueueable async-writes an Agent_Rationale__c + recommended action onto the Decision. Slack Block Kit approval card fires. Approver sees the agent's pick, the deterministic winner, and every eligible action in the module — picks, swaps, or dismisses. Siblings in the same run supersede.
COMPOSE
Novel treatment
No action clears threshold — catalogue gap. CompositionQueueable async-invokes a composer model, drafts channel + offer + content, then re-runs every rail deterministically. Rails-checked drafts land in NBA_Composed_Treatment__c as pending_approval with a full Rails_Checked__c audit trail. Approved compositions can be promoted to first-class catalogued actions.
03
Channel Execution
v8 shipped a pluggable dispatcher framework — four channels live, more via Type.forName probe chain

Every approved decision triggers ExecutionDispatcher. The dispatcher resolves the right channel handler for the winning action's channel, stamps Execution_Status__c and Execution_Error__c on the Decision, and fires the outbound payload. Exploration mode optionally re-weights channel choice using Laplace-smoothed acceptance rates updated nightly.

💬
Slack

Info cards for auto-decisions, approval cards for Tier 2 and compositions, outbound-call briefs for rep actions. Block Kit, interactive buttons, threaded replies. The primary surface for operators.

👤
Sales Cloud

nbaRecommendationPanel LWC on Contact + Account pages. Real-time list of recommended actions with talking points and one-click accept/decline. Outcome writes back to the Decision record.

📧
MC Next Flow

ExecutionMCNextFlowHandler invokes a configurable Autolaunched Flow that triggers the MC Next journey specified on NBAAction__c.ExperienceRef__c. Journey handles templating, rendering, and delivery.

🔗
Webhook

Generic HTTPS POST to any endpoint configured on NBA_Module_Config__mdt. For custom channels, third-party systems, and the "we ship the decision, you deliver it" pattern.

🧪

Exploration mode: a configurable percentage of Tier 1 decisions deliberately re-weight channel or action choice using Thompson sampling over Laplace-smoothed acceptance rates. Nightly batch updates the weights. Contextual bandit behaviour — the engine keeps learning which channels actually work for which segments, rather than always serving the historical winner.

Why the loop matters
On day one, Einstein scores on historical CRM and ERP data — a good starting point, not yet tuned to this org's reps, regions, and contacts. Every accepted action, every declined offer, every null_action restraint row is a training signal. The scheduler refuses to retrain while Identity Resolution is running (IR re-runs spike credits), skips locked CIs, and posts a Block Kit summary on success. After a few retraining cycles the model is meaningfully more accurate than any rules-based approach — and the composition agent has learned which catalogue gaps to fill.
🧭
The Proactive Agentic Layer
The engine reasons about itself daily — not just per-decision. Operator reviews suggestions instead of watching dashboards.

Three queueable agents run nightly, reading the Decision Audit Trace and outcome-chase chains. They notice patterns, propose changes, and drop insights into the unified Review Queue. Nothing self-applies by default — the human approves. Over time, approvals train which insights are worth surfacing.

Model Debug Agent

Watches Einstein scores. Flags score collapse (stddev < 0.05), low-confidence spikes (7d vs 30d), and distribution anomalies (>80% extreme). Drafts plain-language explanations of suspected drift.

Segment Strategy Agent

Parses Decision_Trace__c for high segment-mismatch drop rates and narrow eligibility funnels. Proposes segment tweaks, new excluded-segment links, or catalogue extensions.

Journey Suggestion Agent

Reads outcome-chase chains. When a pattern (e.g. discount → follow-up → escalation) repeats 5+ times with high acceptance, drafts a concrete Flow / MC Next journey spec.

This is the architectural move. Agents that reason about the system, not just per decision. Cannot be retrofitted onto a rules engine — rules are deterministic, reasoning is non-deterministic. Architecturally incompatible unless you design for it from day one. The operator's job shifts from "watch dashboards and adjust" to "review suggestions and approve."
